Case Study

Inline Width Traceability and Operator Monitoring System

Built a production-ready width logging and inspection workflow by acquiring assigned PLC data over EtherNet/IP, logging it through a resilient Windows Service, and extending it into a tray-based operator monitoring and inspection tool.

This project started as a tape-width measurement and logging task, but evolved into a broader production traceability system focused on data reliability, machine-state-aware logging, and operator usability.

System Overview

The width-measurement chain was established from the laser measurement hardware into the Keyence PLC, then extended to PC-side data collection and operator tooling. The final system architecture centered on EtherNet/IP acquisition of assigned PLC values such as motor-running state, width value, and rewinder position, with the PC side split into a background service and a separate tray application.

Final production-oriented architecture used for inline width traceability, background logging, and operator inspection.

Challenge

The production need was not just to save measurement data into CSV. The real challenge was to build a reliable width-data workflow that could capture measurement values from the machine, log them only during active rewinding, survive communication interruptions, and present the results in a form useful both during operation and after the run.

Early work explored native PLC-side logging and FTP transfer in order to understand the machine’s built-in logging behavior and limits. That investigation was useful for learning and validation, but it was not kept as the final architecture.

Engineering Approach

  • Measurement path validation: established the communication chain from the width sensor system into the PLC, confirmed live width data in the PLC, and prepared width, coordinate, direction, and logging-related values.
  • Early logging investigation: tested PLC-side logging and FTP transfer to understand native logging/tracing behavior, file creation, and transfer timing.
  • EtherNet/IP backend: moved to PC-side logging through EtherNet/IP, starting from a minimal read test and evolving it into a reconnect-aware C#/.NET Windows Service.
  • Machine-state-aware logging: assigned and read a maintained motor-running state together with width value and rewinder position so logging occurred only during active operation.
  • Operator tooling: extended the backend into a run-based system with metadata, violations output, live monitoring, and historical inspection through a tray application.

Final Architecture

The final solution used assigned PLC data over EtherNet/IP rather than relying on FTP as the production data path. A background Windows Service handled PLC communication, run detection, logging, and file generation. A separate tray application handled operator setup, live monitoring, and completed-run inspection. The later V2 work also shifted the data model from daily files to per-run structured logging with metadata and violations output.

Operator Interface and Run Inspection

To make the system useful beyond background logging, I extended it into an operator-facing tray application. The interface was designed for two main use cases: live monitoring during active rewinding, and completed-run inspection after logging finished.

What Was Built

Backend / service side

  • PLC EtherNet/IP communication
  • conditional logging based on active machine state
  • automatic reconnect behavior after communication loss
  • status/event logging separate from measurement data
  • run-based file creation, metadata output, and violations output

Operator / inspection side

  • next-run setup fields
  • live monitor
  • live tape diagram
  • historical run inspection
  • marker-based point inspection
  • zoom and pan controls
  • synchronized tape view and width trendline

Result

The project progressed from a basic measurement-logging task into a structured, operator-ready system. The final solution provided reliable EtherNet/IP acquisition, resilient background logging, run-based organization of output files, and an operator-facing interface for both live monitoring and post-run inspection.

More importantly, the project changed the workflow from “collect raw data somehow” to “make width data usable for production decisions, inspection, and traceability.”

Early Investigation Note

An initial FTP-based logging path was intentionally explored first to understand native PLC logging behavior, transfer timing, and practical limits. That phase was a learning and validation step requested early in the work. The final production-oriented solution was built around EtherNet/IP acquisition and PC-side logging instead.

What This Project Demonstrates

This project demonstrates systems integration across industrial sensing, PLC data handling, EtherNet/IP communication, software reliability, deployment thinking, and operator workflow design. It also shows my preferred engineering path: first make the data path work, then make it reliable, then make it usable.
